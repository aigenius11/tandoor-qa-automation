 # Дипломный проект: Автоматизация тестирования Tandoor Recipes

## 1. Описание проекта
Данный проект представляет собой комплекс автотестов для веб-приложения по управлению рецептами Tandoor Recipes. Работа выполнена в рамках дипломного проекта по курсу QA Automation. Основная цель — автоматизация критических сценариев пользовательского пути (UI и API) и настройка процесса непрерывной интеграции (CI).

Технологический стек:
* Язык: Python 3.10
* Фреймворк: Pytest
* Автоматизация UI: Selenium WebDriver
* Паттерн: Page Object Model (POM)
* CI/CD: GitHub Actions
* Отчетность: Allure Framework

## 2. Требования для запуска
Для корректной работы проекта необходимо наличие:
* Python версии 3.10.
* Менеджера пакетов pip.
* Установленного Google Chrome (используется WebDriver Manager для автоматической подгрузки драйвера).

## 3. Инструкции по установке и запуску
1. Клонируйте репозиторий:
git clone https://github.com/aigenius11/tandoor-qa-automation.git
2. Перейдите в папку проекта:
cd qa_auto_tandoor
3. Установите зависимости:
pip install -r requirements.txt
4. Запустите тесты:
pytest --alluredir=allure-results

## 4. Структура проекта
* .github/workflows/ — конфигурация CI/CD для автоматического запуска тестов в GitHub Actions.
* api/ — модули для взаимодействия с API приложения (включая client.py).
* pages/ — реализация паттерна Page Object. Содержит логику работы со страницами (Login, Recipes, Meal Plan, Shopping List).
* tests/ — наборы тестовых сценариев (UI тесты, API тесты, E2E сценарии).
* screenshots/ — скриншоты результатов тестирования и ошибок.
* conftest.py — конфигурационный файл pytest с фикстурами.
* requirements.txt — список всех внешних зависимостей.

## 5. Реализованные сценарии тестирования
Автоматизированы следующие ключевые области:
* Авторизация (UI): валидный вход и проверка перехода в Dashboard.
* API-клиент: проверка взаимодействия с бэкендом через токены.
* Управление рецептами: создание тестовых данных и проверка их отображения.
* Shopping List & Meal Plan: комплексные тесты на добавление продуктов и планирование питания.
* E2E сценарии: сквозные проверки функциональности приложения от логина до результата.

## 6. Список используемых библиотек
* pytest — базовый фреймворк для тестирования.
* selenium — управление браузером и элементами UI.
* webdriver-manager — автоматическая установка ChromeDriver.
* allure-pytest — интеграция с системой отчетности Allure.
* python-dotenv — управление секретными данными через .env.
* requests — для реализации API тестов.

## 7. Инструкция по получению API-токена
1. Авторизуйтесь в приложении Tandoor Recipes.
2. Перейдите в User Settings (Настройки пользователя).
3. Откройте раздел Security или API Keys.
4. Создайте новый ключ (Token), скопируйте его для использования в тестах.

## 8. Пример файла с переменными окружения (.env)
Для локального запуска создайте файл .env в корне проекта:
TANDOOR_USERNAME=ваш_логин
TANDOOR_PASSWORD=ваш_пароль
TANDOOR_API_TOKEN=ваш_токен_из_пункта_7

## 📈 Стратегия развития и расширения тестового покрытия (Roadmap)

Для обеспечения максимальной надежности системы и перехода к полноценному процессу Continuous Testing, определены следующие приоритетные направления развития фреймворка:

### 1. Углубление уровня тест-дизайна
*   Параметризация и Data-Driven Testing: Внедрение библиотеки @pytest.mark.parametrize для проведения кросс-браузерных и кросс-платформенных проверок API с различными наборами входных данных (граничные значения, эквивалентные классы).
*   Комплексное негативное тестирование: Расширение проверок на обработку исключений, включая имитацию истечения срока действия JWT-токенов (401 Unauthorized), попытки IDOR-атак (доступ к чужим ресурсам) и валидацию схем данных при некорректных типах полей.
*   Поиск и фильтрация: Автоматизация сложных сценариев поиска рецептов с использованием комбинаторных фильтров (по тегам, ингредиентам, времени приготовления и калорийности).

### 2. Оптимизация архитектуры и производительности
*   Параллелизация выполнения: Настройка модуля pytest-xdist для распределенного запуска тестов, что позволит значительно сократить время прогона пайплайна в GitHub Actions при масштабировании проекта.
*   Smart Waiting & Stability: Замена статических ожиданий на динамические (Explicit Waits) и внедрение механизма повторных запусков упавших тестов (Retries) для минимизации ложноположительных результатов (*Flaky tests*).

### 3. Интеграция и мониторинг (CI/CD)
*   Гибридная модель тестирования: Реализация сквозных сценариев (E2E), где подготовка данных и удаление артефактов (Cleanup) производятся через API, а ключевые бизнес-действия — через UI. Это обеспечит стабильность пайплайна вне зависимости от динамики фронтенда.
*   Расширенная отчетность: Интеграция уведомлений о результатах сборки в Slack или Telegram с вложенными логами и прямыми ссылками на исторические тренды в Allure Report.
*   Contract Testing: Внедрение проверок на соответствие контрактов API, чтобы исключить поломки при обновлении бэкенда без изменения версии API.

### 4. Автоматизация жизненного цикла Meal Plan
*   Динамическая обработка сущностей: Реализация механизмов парсинга логов или ответов API для автоматического извлечения ID созданных планов питания. Это позволит полностью автоматизировать жизненный цикл Meal Plan (создание -> просмотр -> редактирование -> удаление) без необходимости ручного вмешательства.
